## Principle

Use `sm` for dense secondary actions, `md` for the default operational interface, and `lg` only when touch comfort or emphasis requires it.

## Choice guide

- Keep controls in one action row on the same size recipe unless a primary action deliberately needs stronger emphasis.
- Use `sm` where density matters and a pointer is expected, `md` as the default, and `lg` for touch-first or high-emphasis tasks.
- Change the whole recipe—height, padding, gap, font size, and line height—rather than overriding only height.

## States and touch targets

Use semantic component states instead of styling snapshots. Hover previews availability, pressed shows the active press, selected persists a choice, disabled removes an unavailable action, and focus remains visible for keyboard navigation. Do not use hover as the only cue. For touch-first layouts, choose `lg` or add spacing around controls without shrinking the interactive box.

## Implementation

Override the global metric when an entire product needs a different density, or a `--tr-*` recipe token when one component needs a deliberate exception.

```css
:root {
  --tinyrack-control-height-md: 2.5rem;
}

.rack-toolbar {
  --tr-btn-height: 2.25rem;
}
```

## Accessibility

Every control needs a visible label or accessible name, a visible focus state, and a state that does not rely on color alone. Preserve the component's keyboard behavior and use `lg` when a touch-first workflow needs a larger target; do not enlarge only the visual surface while leaving the interactive hit area behind.
