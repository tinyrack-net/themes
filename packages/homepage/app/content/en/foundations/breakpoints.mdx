## Scale

Breakpoint tokens define where a layout gains available space. Start mobile-first and add a <TRCode>{'*-min'}</TRCode> rule only when the content needs it.

<BreakpointReference locale="en" />

## Decision guide

- Add a breakpoint when the content no longer fits its task, not when a named device begins.
- Use viewport breakpoints for page shells and application regions. Keep reusable component adaptations local to the component or its container.
- Prefer CSS for visual reflow. Read the shared value in TypeScript only when behavior, data loading, or an interaction mode must change.

## TypeScript

```ts
import { tinyrackBreakpoints } from '@tinyrack/ui/core';

const desktopQuery = `(width >= ${tinyrackBreakpoints.lg})`;
```

## CSS and Tailwind

Tinyrack CSS uses Tailwind's <TRCode>@variant</TRCode> directive. Import <TRCode>@tinyrack/ui/core.css</TRCode> before component styles so Tailwind resolves the shared breakpoint scale.

```css
@variant md {
  /* Tinyrack source authoring */
}
```

Tailwind uses the same values through <TRCode>--breakpoint-xs</TRCode> to <TRCode>--breakpoint-xl</TRCode>.

```tsx
<section className="grid gap-4 md:grid-cols-2 max-md:grid-cols-1" />
```

Use the generated <TRCode>xs:</TRCode> through <TRCode>xl:</TRCode> variants for mobile-first additions and <TRCode>max-md:</TRCode>-style variants when a rule applies below a threshold. The source of truth is <TRCode>--breakpoint-*</TRCode> in CSS and <TRCode>tinyrackBreakpoints</TRCode> in TypeScript.

## Responsive QA

Check each changed layout immediately below, at, and above its breakpoint. Include a 390px mobile viewport, 200% zoom, long localized labels, keyboard focus order, and confirmation that no essential action disappears when the layout collapses.
