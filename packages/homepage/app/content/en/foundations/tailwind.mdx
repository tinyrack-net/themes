<h2 className="tr-mdx-h2" id="setup">Setup</h2>

Import Tailwind first, then Tinyrack core CSS. The core stylesheet keeps the runtime <TRCode>--tinyrack-*</TRCode> variables and exposes them through Tailwind v4's static theme namespaces.

```css
@import "tailwindcss";
@import "@tinyrack/ui/core.css";
```

Component styles remain separate and should be imported only when used.

```css
@import "@tinyrack/ui/components/button.css";
```

<h2 className="tr-mdx-h2" id="choose-utilities">Choose the right utility</h2>

- Use a <TRCode>tinyrack-*</TRCode> utility when color, type, spacing, shape, elevation, or motion expresses a design-system decision.
- Use the shared <TRCode>xs:</TRCode> through <TRCode>xl:</TRCode> responsive variants for viewport thresholds.
- Keep Tailwind defaults for incidental layout that does not represent a Tinyrack token, such as a one-off grid column count.
- Do not override other Tailwind defaults globally. The prefix keeps existing application code and third-party examples predictable.

```tsx
<section className="grid gap-tinyrack-lg rounded-tinyrack-lg border-tinyrack-default border-tinyrack-border bg-tinyrack-surface p-tinyrack-xl">
  <h2 className="text-tinyrack-xl font-tinyrack-heading">Cluster health</h2>
  <p className="text-tinyrack-sm leading-tinyrack-md text-tinyrack-text-muted">
    All nodes are available.
  </p>
</section>
```

<h2 className="tr-mdx-h2" id="arbitrary-values">Arbitrary values</h2>

Use a type hint when a component-scoped <TRCode>--tr-*</TRCode> variable or another CSS value has no natural Tailwind theme namespace. Keep the variable as the source of truth instead of copying its resolved value.

```tsx
<div className="bg-[color:var(--tr-card-background)] [width:var(--tr-layer-anchor-width)]">
  ...
</div>
```

Prefer the named utility in the reference below whenever one exists. Arbitrary values are for deliberate component customization, not an alternate token vocabulary.
